## 1. プロジェクト誕生

CRDebugger の物語は、ある一つの問題意識から始まっている。.NET アプリケーションに「クラッシュレポーター」と「デバッガ UI」をプラグインとして組み込む際、開発者は毎回ゼロから実装しなければならない。バグレポートの送信、プロファイリング、ログコンソール、オプション管理——これらを WinForms でも WPF でも Avalonia でも動く NuGet パッケージとして提供できないか。

そのビジョンが形になったのが CRDebugger だ。アーキテクチャの中核となる発想は「ソースリンク方式」という、やや珍しい設計である。`CRDebugger.Core` はパッケージとして公開せず、各プラットフォームプロジェクト（WinForms / WPF / Avalonia）が Core の `.cs` ファイルを `<Compile Include>` で直接コンパイルして単一 DLL として提供する。利用者は `dotnet add package CRDebugger.Avalonia` の一行だけで全機能を使える。推移的依存が発生しないクリーンな体験だ。

タイムラインが記録されている 2026 年 3 月 27 日時点では、すでにバージョン 1.0.x のリリースサイクルに入っており、その日だけで v1.0.16 から v1.0.18 までバンプするという密度の高い一日となった。

## 2. アーキテクチャの進化

### 初期設計：ソースリンク戦略の確立

観察 #472〜#477（2026-03-27 10:28–10:30）は、プロジェクト構造を俯瞰した最初のスナップショットを記録している。この時点で既に 4 プロジェクト構成が完成していた：

```
CRDebugger.Core     (IsPackable=false)
CRDebugger.WinForms ← Core ソース内包
CRDebugger.Wpf      ← Core ソース内包
CRDebugger.Avalonia ← Core ソース内包 + クロスプラットフォーム
```

### .NET 10.0 マルチターゲティング対応（S150）

最初の大きな変更点は、セッション S150 でトリガーされた .NET 10.0 対応だ。観察 #478〜#481 が示すように、4 プロジェクト全てに `net10.0`（および `net10.0-windows`）を追加した。

しかし単純ではなかった。観察 #482 が記録した通り、`net10.0-windows` ターゲットで WinForms Designer の Serialization 要件によるビルドエラー `WFO1000` が発生。観察 #484 で `DesignerSerializationVisibility` 属性を `IsSelected` プロパティに追加して解決した。

また観察 #493・#495 では、テストプロジェクトの `Microsoft.NET.Test.Sdk 18.3.0` が .NET 6.0 と非互換であることが判明し、テストプロジェクトを .NET 8.0 へアップグレードする対応を取った。マルチターゲティングの追加が思わぬ連鎖反応を引き起こすというよくあるパターンだ。

### Timer 曖昧参照問題

CLAUDE.md に明示的に記載されている「WinForms では `System.Windows.Forms.Timer` と `System.Threading.Timer` が衝突する」という問題も、実際の開発過程での学習から生まれた知識だ。Core 内では完全修飾名 `System.Threading.Timer` で記述することで解決している。この種の「ハマりポイント」が CLAUDE.md に蓄積されていくことで、次のセッションで同じ過ちを繰り返さない仕組みが機能している。

## 3. 主要なブレークスルー

### ControlTheme による FluentTheme 完全バイパス（S159）

Avalonia の FluentTheme は `ToggleButton` や `Button` がシステムアクセントカラーを使う設計になっている。スタイルセレクタや Resource オーバーライドでは解決できないという壁にぶつかった後（S157, S158 で試行錯誤）、観察 #565〜#570（2026-03-27 14:04–14:06）が記録した S159 でブレークスルーが訪れた。

**解決策：`ControlTheme` でテンプレートごと差し替える**。`FilterToggleTheme` と `SidebarButtonTheme` という独自テーマを作成し、FluentTheme の介入を根本から排除した。これは単なる色の上書きではなく、コントロールのビジュアルツリー全体を自前で定義するという大胆なアプローチだ。

```
❌ スタイルセレクタ → FluentTheme が勝つ
❌ Resourceオーバーライド → 部分的にしか効かない
✅ ControlTheme でテンプレート差し替え → 完全制御
```

### Win32 DWM API 統合（S165）

観察 #583（2026-03-27 14:58）が記録したもう一つのブレークスルーは、Win32 DWM API を直接呼び出してウィンドウクロームを制御することだ。`DWMWA_BORDER_COLOR`、`DWMWA_CAPTION_COLOR`、`DWMWA_USE_IMMERSIVE_DARK_MODE` を使い、OS のアクセントカラーがタイトルバーに流入するのを強制的に遮断した。

さらにタイトルバーのアイコン非表示には `WS_EX_DLGMODALFRAME` フラグと `WM_SETICON` メッセージの組み合わせを使った（観察 #665, #666, #667）。これらは「Avalonia の AXAML だけでは解決できない」問題に対して、Win32 レイヤーに踏み込むことで決着をつけた事例だ。

## 5. 技術的負債

### 半透明色の誘惑

最大の技術的負債は、まさに黄色UI問題の根本原因でもある「半透明色の使用」だった。`#FFFFFF06` のような半透明白色は「ほんのり明るくなる」効果を意図したものだが、Avalonia の FluentTheme が `SystemAccentColor`（ユーザーの OS 設定）を背景として流入させる仕様と組み合わさると、黄色やピンクなど意図しない色が現れる。

観察 #632〜#655 は、S170 セッションでの系統的な返済の全記録だ。`SharedStyles.axaml`、`ConsoleView.axaml`、`BugReporterView.axaml`、`OptionsView.axaml` 内の半透明色を一件ずつ固定色に置換している。合計 24 件の bugfix 観察がこの清算に要したコストを物語っている。

### `ExtendClientAreaToDecorationsHint` の罠

アクリルブラー効果を実装するために `ExtendClientAreaToDecorationsHint="True"` を使っていたが（観察 #489 が記録）、これは DWM のタイトルバー背景（= SystemAccentColor）をクライアントエリアに流入させるという副作用があった。観察 #574 でアクリルブラーを完全撤廃し、観察 #584 で Win32 DWM API 統合という代替手段に置き換えた。近道を取って後で返済したパターンの典型だ。

## 6. 課題とデバッグサーガ：黄色UI問題との多セッション格闘

黄色UI問題は、この開発史の中で最も劇的な章だ。問題の発覚から解決まで、実に **7 つのセッション（S151〜S165, S170）** にまたがり、約 5 時間（10:31〜15:04）を費やした。

### ラウンド 1：スタイルセレクタ戦（S151〜S152, 10:31〜10:38）

観察 #550 が記録した「Avalonia Styling Capabilities and Patterns」（114,215 トークン消費！）は、問題理解のための情報収集フェーズだ。最初の試みは `ListBoxItem` のスタイルオーバーライドで、`SharedStyles.axaml` にカスタムスタイルを追加した（観察 #490）。しかし FluentTheme の詳細度に負けて効果は限定的だった。

### ラウンド 2：Resource オーバーライド戦（S157〜S158, 11:25〜13:02）

観察 #554〜#556 でトグルボタンとサイドバーボタンのフォーカス状態オーバーライドを追加し、観察 #561〜#563 で `SystemAccentColor` 自体を Resource で上書きするアプローチを試みた。改善はされたが完全解決には至らなかった。

### ラウンド 3：ControlTheme 完全置換（S159, 13:05〜14:06）

観察 #565 からの一連の変更が真の転換点だ。`FilterToggleTheme` と `SidebarButtonTheme` を新設し、FluentTheme の ToggleButton/Button テンプレートを完全にバイパスした。しかしまだウィンドウ枠の黄色が残っていた。

### ラウンド 4：ウィンドウ背景の模索（S160〜S164, 14:07〜14:47）

観察 #572 でアプリケーションレベルのアクセントカラー上書き、観察 #573 でウィンドウ背景を透明から不透明に変更、観察 #574 でアクリルブラー完全撤廃……各施策が次の問題を露わにする「叩けばホコリ」状態が続く。さらに XAML の閉じタグミスによるビルドエラーが 3 連続で発生し（観察 #575, #577, #578）、デバッグ中にデバッグが必要という踏んだり蹴ったりの展開もあった。

### ラウンド 5：Win32 DWM API による根本解決（S165, 14:51〜15:03）

観察 #582〜#591 が記録した最終ラウンド。Git 分析でどのコミットから問題が出始めたかを特定し（観察 #579〜#581 のブランディングテストと元に戻す作業）、Win32 DWM API を直接叩いてタイトルバーの色を強制指定することで枠線の黄色を撃退。同時に残存する半透明色を全て固定色に変換した。

### ラウンド 6：全色の系統的清算（S170, 15:04〜15:48）

観察 #632〜#656 の 25 件は、プロジェクト全体に残存していた半透明色を一件ずつ発見・修正する「地雷除去」作業の記録だ。そして最後の観察 #656「Verified Yellow UI Tinting Fix with Successful Clean Build」は **132,232 トークン** を消費した。この一件だけでプロジェクト全体のトークン消費の約 9% を占める。

**黄色UI問題が CLAUDE.md に「鉄則」として記録されたのは、これだけの代償を払った教訓だからだ。**

## 10. 教訓とメタ観察

### 教訓 1：Avalonia の FluentTheme は「乗りこなす」より「バイパスする」

黄色UI問題に費やした 5 時間と 7 セッションが証明している。Avalonia の FluentTheme はカスタマイズを前提にした設計ではなく、半透明色・`ExtendClientAreaToDecorationsHint`・アクセントカラー依存のリソースと組み合わせると予測不能な挙動を示す。解決策は「テーマのルールに従おうとする」ことではなく、`ControlTheme` でテンプレートを完全に置き換えて支配権を取り戻すことだった。

### 教訓 2：半透明色は「後払いの負債」

`#FFFFFF06` のような薄い半透明色は、開発時には問題が見えにくい。しかし `SystemAccentColor` という不確定な変数が介在すると、ユーザーの環境によって全く異なる色に見える。観察 #632〜#655 の系統的な修正作業が示す通り、初期設計で不透明色を徹底していれば約 24 件分のデバッグ作業は不要だった。

### 教訓 3：「動いているプロセスからファイルロック」は見えにくいブロッカー

観察 #498 が記録した「Build blocked by file lock from running Sample.Avalonia application」は、開発中に頻繁に遭遇するが見落としやすい問題だ。

### 教訓 4：WPF XAML での `assembly=` 指定は禁物

ソースリンク方式では Core の型が各プラットフォームの DLL に内包されるため、`assembly=CRDebugger.Core` という指定が実行時エラーになる。CLAUDE.md に記録された「assembly= を省略すること」というルールは、ソースリンク設計の最も反直感的な側面に対応した知識だ。

### 教訓 5：変更ごとにビルド確認

観察 #575〜#578 の3連続 XML タグミスエラーは、大きな変更（アクリルブラー除去）の後に細かいミスが積み重なって発生した。変更を細かく分割しビルドを頻繁に確認するパターンの重要性を物語っている。

### メタ観察：永続メモリが開発スタイルを変える

S175（観察 #670）で黄色UI問題の解決後すぐに CLAUDE.md に Best Practices を追記したことが示す通り、永続メモリがあることで「次のセッションで使えるように記録する」という動機が自然に生まれる。これは個人開発者の知識管理に大きな変革をもたらす可能性がある。

### メタ観察：1 日 211 観察という密度が意味するもの

約 9.5 時間で 211 件の観察が生成された（平均 22 件/時間、約 2.7 分に 1 件）。discovery → change → bugfix → feature という観察タイプの連鎖が、問題解決の思考プロセスをそのまま記録した地図になっている。
